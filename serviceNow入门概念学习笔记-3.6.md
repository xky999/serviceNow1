### 5.核心应用管理议程Core Application Administration Agenda

#### 5.2应用安全（Application Security）

​	系统管理员通常使用三个安全模块：

​		**系统属性 > 安全** 是管理员用于设置高级安全选项的应用模块。例如，你可以在此处设置当表（Table）上没有访问控制列表（ACL）时的安全管理器默认行为（显式设置拒绝访问或允许访问）。

​		**系统安全 > 访问控制** 用于管理访问控制规则（Access Control Rules）和列表。

​		**系统安全 > 高安全设置** 是管理员用于设置更严格安全选项的模块。

​	访问控制（Access Control）：访问控制是一条安全规则，在**行级别**（对记录（Record）的访问）和**列级别**（对字段（Field）的访问）定义和设置，并在尝试访问任何 ServiceNow 表（Table）时执行

​	访问控制列表（**ACL**）是一个表的所有访问控制的列表

​	访问控制由角色（Roles）、条件表达式（conditional expressions）和脚本（scripts）定义

​		**角色（Roles）**：在访问控制规则的中间有一个角色部分。如果你在那里添加一个或多个角色，那么只有拥有至少一个这些角色的用户才被允许执行所请求的操作。

​		**条件表达式（Conditional Expressions）**：在访问控制规则的中间，你会看到一个条件部件（Widget），你可以在其中为 ACL 添加条件表达式。例如，如果你希望此规则仅对数据库记录评估为真，你可以添加一个条件表达式 “category is Database”。

​		**脚本（Scripts）**：如果勾选了 “高级（Advanced）” 复选框，那么你就有机会基于用户定义的脚本应用安全控制。你的脚本可以访问当前记录（Record），并负责设置全局 “answer” 变量，以允许或拒绝访问所请求的资源 / 操作。脚本会与你在规则上设置的其他条件一起评估。所有条件都必须评估为真。

​	实例（Instance）使用访问控制列表（ACL）规则来控制用户可以访问哪些数据以及如何访问这些数据

​	每个 ACL 规则指定：被保护的对象,访问该对象所需的权限

​	ACLs provide:

​		一种全局安全机制，用于限制对 ServiceNow 表记录的行和列的读取、写入、创建和删除操作。

​		通过定义不同类型的组件来评估为真，从而实现内置的灵活性。

​		通过上下文安全管理器（Contextual Security Manager）实现更轻松的管理，因此在添加或更改 ACL 时，可以感知到所有 ACL。

​	

访问控制工作流（Access Control Workflow）

​	上下文安全（Contextual Security）最强大的特性之一是，它既可以应用于特定字段（Field），也可以应用于整行（Row）

​	记录 ACL 规则按以下顺序处理：

​		将对象与字段 ACL 规则进行匹配。

​		将对象与表（Table）ACL 规则进行匹配。

​	这种处理顺序确保用户在访问更通用的对象之前，先获得对更具体对象的访问权限。用户必须同时通过字段和表 ACL 规则，才能访问一个记录（Record）对象。

​		如果用户未通过字段 ACL 规则，但通过了表 ACL 规则，则拒绝用户访问该字段 ACL 规则所描述的字段。

​		如果用户未通过表 ACL 规则，则即使用户之前通过了字段 ACL 规则，也会被拒绝访问该表中的所有字段。

高安全设置（High Security Settings）

​	安全管理（Security Administration）：

​		将安全设置整合到一个简化的界面中。

​		新增安全管理员角色（security_admin）。

​		引入提权（elevated privilege）概念。

​	ACL 修改（ACL Modifications）：

​		新增关键系统级 ACL。

​		属性访问控制（Property Access Control）。

#### 5.3 通知（Notifications）

什么是**事件（Events）**和**通知（Notifications）**？

​	事件是向 ServiceNow 进程指示某事已发生的信号

​	事件由以下原因触发：

​		用户操作：登录、批准请求、重命名附件

​		脚本：业务规则（Business Rules）、工作流（Workflow）

​	通知可以由平台中的事件触发，但与业务规则不同，通知无需脚本知识，也可以在记录（Record）插入或更新时创建

​	事件 / 通知流程（Event/Notification Flow）

​	事件生成器（Event Generators）： 脚本（Script）/业务规则（Business Rule）/工作流（Workflow）——>事件队列（Event Queue，生成事件的记录）-->脚本操作（Script Actions）/电子邮件通知（Email Notifications）

​	事件可以通过多种方式生成。最常见的生成方式是使用业务规则或工作流活动（Workflow Activities）。

​	必须对事件做出响应，才能触发后续动作：

​		脚本操作是响应生成事件的服务器端脚本。脚本操作拥有 ServiceNow JavaScript API 的全部能力。

​		电子邮件通知（或短信、会议邀请）用于向用户发送消息。



### 6 持续维护议程（ Ongoing Maintenance Agenda）

#### 6.1 更新集（Update Sets）

​	更新集是一组可从一个实例（Instance）迁移到另一个实例的自定义配置

​	更新集允许管理员将一系列变更分组到一个命名的集合中，然后将它们作为一个整体迁移到其他系统

​	**如何使用更新集：**

​		更新集用于将自定义配置从一个实例迁移到另一个实例

​		允许在开发实例中创建和修改自定义配置，然后应用到生产实例

​	**更新集中会捕获哪些内容？**

​		**被捕获（流程记录）**

​		业务规则（Business Rules）

​		客户端脚本（Client Scripts）

​		字段（Fields）

​		表单及表单分区（Forms and Form Sections）

​		报表（Reports）

​		表（Tables）

​		视图（Views）

​		工作流（Workflows）

​	**不被捕获（数据）**

​		修改后的配置项（Modified CIs）

​		修改后的数据记录（Modified Data Records）

​		新数据记录（New Data Records）

​		新用户和用户组（New Users and Groups）

​		计划（Schedules）

​		计划任务（Scheduled Jobs）

​	主页必须手动添加到更新集

更新集的比较、回滚与合并

​	变更会被跟踪，你可以通过 `update_synch` 属性自定义表中的对象：

​		表（Tables）

​		表单（Forms）

​		字段（Fields）

​		业务规则（Business Rules）

​		客户端脚本（Client Scripts）

​		视图（Views）

​	回滚已比较的版本（Revert Compared Versions）

​		可以在回滚变更前比较版本

​		无法回滚数据库变更

​		可以回滚到最新的基础版本

​		无法回滚到更早的基础版本

​	回滚变更的步骤：

​		查看对象的版本列表。

​		右键单击某个版本，选择 “回滚到此版本（Revert to this version）”。

​		点击 “确定（OK）” 确认操作。

​		所选版本将成为当前版本。

​	合并更新集（Merge Update Sets）

​		更新集可以被合并

​		如果两个用户在不同的更新集上工作，它们可以被合并为一个更新集，以便于传输

​	这为哪些内容被迁移、哪些没有被迁移提供了参考。在合并和验证后，建议删除或清空原始更新集。系统不会从更新集中移除更新，除非该更新是合并时被选中的那一个。

​	**有效管理变更与沟通**

​		制定计划并确定统一的迁移路径

​		了解正在开发的内容，并确保管理员知晓开发进展

​	**在一个更新集中包含多项变更**

​		避免：“变更多，更新集也多”

​		将同类项目分组到一个小型、可管理的集合中

​		在迁移更新集前使用 “预览（Preview）” 功能，比较潜在冲突

#### 6.2 升级与性能（Upgrades and Performance）

​	ServiceNow 发布周期（ServiceNow Release Cycle）

​	ServiceNow 发布周期旨在提供最佳的稳定性和质量，同时具备快速解决问题和交付新功能的灵活性。

​	ServiceNow 对特定功能集使用 “系列（family）” 的概念。

​	一个系列包含：

​		**功能发布（Feature releases）**：包含新功能和对现有功能的修复。

​		**补丁发布和热修复（Patch releases and hotfixes）**：提供问题修复，并根据需要发布。

​	**季度补丁计划（Quarterly Patching Program, QPP）**

​		季度（Quarterly）

​			ServiceNow 季度补丁计划（QPP）会在全年内按季度定期对客户实例进行补丁更新。

​			该计划的目标是确保所有客户实例都获得最新的安全、性能和可用性更新。

​		发布系列（Release Families）

​			处于受支持发布系列中的实例，会被补丁更新到目标版本。

​			QPP 不会将实例升级到新的发布系列，而是在同一系列内进行补丁更新；唯一例外是，如果客户使用的是不受支持的系列，他们将被升级到新的系列。

​		更新记录（Update Record）

ServiceNow 实施了更新记录功能，支持客户与 ServiceNow 之间的双向私密沟通，以便就 QPP 计划提出任何问题或意见。

​		**QPP 与升级的区别**

与通常旨在提供增强或新增功能的升级不同，QPP 专注于确保客户实例在可用性、性能和安全性方面得到优化。

- **升级**：指从一个发布系列迁移到另一个发布系列。
- **QPP**：专注于同一系列内的补丁更新。虽然可能需要进行必要的升级，但这些升级会作为单独流程执行，不属于季度补丁计划。在 QPP 中，客户、合作伙伴和内部实例会被补丁更新到同一受支持发布系列中的最新版本。
- **注意**：补丁通常不会 “破坏” 实例，也不需要进行广泛测试。



​	**升级中哪些内容会被更新，哪些不会？**

​	升级中豁免的自定义配置

​		任何由客户从基础状态修改过的记录，都会在升级中被**豁免**。

​		这可以防止升级覆盖或破坏用户的自定义配置。

​	被跟踪的自定义配置

​		在客户更新表 `[sys_update_xml]` 中存在更新记录的任何对象，在升级过程中会被**自动跳过**。

​	**跟踪机制**：当一个对象被自定义时，对应的记录会被添加到客户更新表 `[sys_update_xml]`。该表维护所有被自定义对象的当前版本信息。

​	**保护机制**：为了防止自定义配置被系统升级覆盖或破坏，升级过程会自动跳过对在客户更新表中存在当前版本的对象的修改。



​	**性能故障排除**

​	ServiceNow 实例的感知性能由四个核心组件构成：

- 应用服务器响应（Application Server response）应用服务器处理用户请求并渲染结果页面所花费的时间。
- 网络延迟与吞吐量（Network latency and throughput）网络将用户请求传递到服务器，并将响应返回给用户所花费的时间。
- 浏览器渲染与解析（Browser rendering and parsing）用户浏览器渲染 HTML 内容，并解析 / 执行 JavaScript 代码所花费的时间。
- 实例缓存（Instance Cache）系统可用于处理请求的可用资源总量。



#### 6.3 协作（Collaboration）

​	实时编辑（Real-Time Editing）

- 实时编辑记录，并查看其他用户保存的编辑内容（类似 Google Docs），从而提升协作效率。
- 实时编辑是 Presence 功能的扩展，允许多人同时处理同一条记录，显示对方状态（编辑或查看）及编辑内容（通过蓝色 “脉冲” 图标展示）。
- Connect 提供对编辑内容的实时可见性，确保所有人采取正确行动，加速工作完成。



​	活动流内联编辑（Activity Stream Inline Editing）

- 在活动流中直接参与实际工作，无需深入记录本身，更快完成任务。
- 活动流中的内联评论功能，可在更新发生时对活跃记录进行批注，支持同时处理多项工作，提升效率。



​	Connect 的三大**主要用途**

​		点对点聊天（Peer to Peer Chat）

​			集成了 Presence 状态功能，但聊天内容不会自动发布到事件或任务记录中。

​			支持直接的一对一对话或群组对话。

​		 Connect 支持（Connect Support）

​			通过独立插件启用，提供支持队列，由坐席管理，整个聊天会话会被记录到相关任务中。

​			是一个实时消息工具，帮助支持坐席轻松跟踪支持案例、快速查找解决方案并解决问题。

​			管理员可创建聊天队列，让用户访问实时支持；坐席可监控队列提供即时支持；支持拖放分享链接、文件和记录。

​		关注记录（Follow a Record）

​			关注一条记录，以类似聊天的界面显示该记录的所有更新。

​			所有对话消息都会发布到该记录的评论或工作笔记中。
