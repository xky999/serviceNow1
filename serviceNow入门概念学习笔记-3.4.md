## 二、重点记录 

### 1.用户管理

​	**roles：**

​		**系统管理员（System Admin）：**拥有系统所有功能、应用、操作及数据的访问权限。

​		**专业管理员（Specialized Admin）：**管理特定应用的角色，拥有该应用的专属管理权限

​		**处理者（Fulfiller）(ITIL）：**执行与 ITIL 工作流相关的所有活动，完成工单处理

​		**审批者（Approvers）：**可执行请求者的所有操作，且能查看或修改定向给自己的审批请求

​		**请求者（Requester）（ESS）:**无关联角色，仅能提交请求、管理自身请求并访问公共页面

​	ServiceNow 预置角色遵循 “**最小权限**” 与 “**职责分离**” 原则



### 2. 数据管理与报表

#### 2.1 表

​	表的继承：

​		扩展其他表的表称为**子类**

​		被扩展的表称为**父类**

​		如果一份表被扩展，但自身不扩展任何表，则被称为**基类**	



​	对于子类记录，数据库会：

​		储存子类独有的新字段。

​		储存父类中的其余字段。

​		子类中的一条记录同时存在于子类和父类中，共享同一个 `sys_id`。

​		平台会将它们作为一条记录统一访问。

​		当一个类中的记录被删除时，平台会自动删除其匹配类中的对应记录。

​	

​	表关系：

​		一对多：在一个表中，某个字段可以持有对另一个表中记录的引用。

​			ServiceNow 提供了三种一对多关系字段：

​				引用字段：允许用户在引用字段所定义的表上选择一条记录。

​				Glide List ：允许用户在 Glide List 定义的表上选择多条记录。

​				文档 ID 字段：允许用户在实例中的任意表上选择一条记录。

#### 2.2导入集合

​	核心组件说明：

​		**导入集文件类型（Import Set File Types）**：可通过提供路径和认证信息，从本地源或网络服务器导入文件（支持 CSV、Excel、XML 等格式）。

​		**导入集表（Import Set Table）**：作为从数据源导入记录的**暂存区**，用于临时存储待转换的数据。

​		**转换映射匹配字段（Transform Map Match Fields）**：转换映射定义了导入集表字段与现有 ServiceNow 表字段之间的关系，指导数据如何映射。

​		**目标表（Target Tables）**：数据转换完成后，最终被写入的 ServiceNow 业务表（如用户表、配置项表）。



​		**转换映射（Transform Maps）**：转换映射提供了将数据从**导入集表**迁移到**生产表**的指南，字段映射直接指定数据的流向。

​		转换映射非常灵活：

​			可以简单到让应用自动匹配源和目标的字段名；

​			也可以使用高级逻辑，充分利用 ServiceNow 的脚本环境。

​			单个导入集字段可以映射到生产表中的多个字段。

​			任何表都可以作为导入集转换的目标，表中的任何字段都可以作为导入集字段转换的潜在目标。

​		**关键操作提示：**

​			**分批导入**：避免一次性导入大量数据，建议分批处理，防止系统性能下降。

​			**先测试再生产**：在生产环境执行导入前，务必在子实例中完整测试转换映射和数据校验逻辑。

​			**字段校验**：使用转换映射中的脚本（如 `onBefore`、`onAfter`）进行数据清洗和校验，确保导入数据的质量。

​		**自动映射工具（Automatic Mapping Utility）：**最简单的映射方法是，当导入集表的所有字段名与生产表（数据将被转换到的表）的字段名完全匹配时使用。在这种情况下，只需在表转换映射表单的相关列表中点击 “自动匹配（Auto Map Matching Fields）”，即可确认正确的匹配。如果在自动匹配过程中出现字段匹配方式的差异，可以使用映射辅助工具轻松纠正。当所有字段都正确匹配后，单击 “转换（Transform）”，开始将数据转换到目标表。

​		**映射辅助工具（Mapping Assist Utility）：**映射辅助工具提供了一个直观的可视化环境，用于指定导入集字段与生产表字段之间的映射关系。使用该工具，可以将单个源字段（导入集表上的字段）映射到多个目标字段（生产表上的字段）。

​	

​	**合并字段：**在数据导入中，对一个（或一组）字段进行**合并（coalescing）**，意味着该字段将被用作**唯一键（unique key）**：

​		如果通过合并字段找到匹配项，则使用导入的信息**更新**现有记录；

​		如果未找到匹配项，则在数据库中**插入**一条新记录。

​	**合并配置方式：**

​		**无合并（No coalesce）：**如果未定义合并字段，所有导入行都将被视为新记录，不会更新任何现有记录。

​		**单字段合并（Single-field coalesce）：**可以在单个字段上进行合并，以更新现有记录。如果目标表中存在一条记录，其合并字段的值与暂存表记录中的值相同，则使用导入集记录的值更新目标表记录。

​		**多字段合并（Multiple-field coalesce）：**可以在多个字段上进行合并，以更新现有记录。如果目标表中存在一条记录，其所有合并字段的值都与暂存表记录中的对应值相同，则使用暂存表记录的值更新目标表记录。目标表和暂存表之间的所有合并字段值都必须匹配，才能进行多字段合并。

​		**条件合并（Conditional coalesce）：**可以使用 ** 脚本（script）** 来决定一条暂存表行是否应合并到目标记录。大多数条件合并脚本都定义在字段映射的 “源脚本（Source script）” 字段中，用于 `sys_id` 字段。要使用暂存表记录的值更新目标记录，脚本必须返回目标表记录的 `sys_id`。

​			

#### 3.3：CMDB

​	**配置管理数据库（CMDB**）应用是一系列表和字段的集合，它包含了公司管控的所有配置项（CIs），以及这些配置项的属性和相互关系。

​	**核心概念：**

​		**CMDB（配置管理数据库）：**ServiceNow 中存储企业 IT 资产（配置项）及其关系的核心数据库，是 IT 服务管理的基石。

​		**CI（配置项）：**企业 IT 环境中可管理的组件，如服务器、数据库、网络设备、应用服务等。

​		**CI 关系：**配置项之间的依赖或关联关系（如 “应用 A 运行在服务器 B 上”），是构建服务依赖图的关键。

​		**逻辑模型：**CMDB 不仅存储孤立的 CI 信息，还通过关系构建出企业 IT 基础设施的完整逻辑视图。



​	**依赖视图（Dependency View）：**

​		依赖视图以图形化方式展示一个配置项（CI）的基础设施视图，以及它所归属和所支撑的业务服务。

​		依赖视图可以指示配置项的状态，并允许访问与该 CI 相关的告警、事件、问题、变更和业务服务。

​		由依赖视图生成的地图基于 D3 和 Angular 技术，提供了现代化的交互式图形界面，用于可视化配置项及其关系。

​		可以使用依赖视图查看为该邮件服务提供数据的其他 “上游” 配置项，也可以查看 “下游”，即该邮件服务所依赖的所有其他配置项（CI）。

​	**核心概念：**

​		上游（Upstream）：为当前服务或 CI 提供数据或支持的其他 CIs（例如，为邮件服务提供数据库支持的服务器）。

​		下游（Downstream）：当前服务或 CI 所依赖的其他 CIs（例如，邮件服务运行所依赖的操作系统和网络设备）。

​	

​	**CMDB 实施考量（CMDB Considerations）**：

​		作用：

​			帮助定位失败的变更及其关联的事件

​			促进快速的影响分析，帮助减少或消除停机时间

​	实施问题（Implementation Questions）：

​		数据如何录入或导入，以及如何进行管理？需要从人员、流程和技术三个维度进行考量。

​		数据存储在何处？识别并相应地扩展相关表。

​		哪些数据是必需的？只存储正在使用的数据，忽略未使用的数据字段。

​		配置项（CI）的导入或更新应在何时进行？规划导入的排期，以保持数据的实时性和准确性。
