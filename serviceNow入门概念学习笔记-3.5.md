### 4. Service Automation Platform

#### 4.1 Knowledge Management

​	层级结构：

​		知识库中的知识文章按类别分组。

​		知识管理员可以定义知识类别，以预填充可用类别列表；知识贡献者可以选择类别，并且如果启用了权限，还可以为知识库添加或编辑类别。

​	结构：

​		管理员可以创建多个知识库，并将其分配给各个managers，由他们负责控制每个知识库的行为和组织方案。

​		管理员创建多个知识库时，它们可以拥有独立的生命周期工作流、用户条件、类别结构和管理分配。

​	安全：

​		用户条件（User Criteria）可以在知识发布之间定义和重用，并且是可扩展的。访问权限还可以按用户组、角色、公司、位置和部门进行额外评估。

​		拥有至少一个角色的用户可以创建和发布知识文章，而没有任何角色的用户可以阅读文章并提交反馈，但不能创建或编辑文章。

#### 4.2 Service Catalog

​	核心组件：

​		**记录生成器（Record Producers）**：构建与请求相关联的系统组件。

​		**目录项（Items）**：服务目录的基础构建块。

​		**变量（Variables）**：提供问题，帮助请求者明确要订购的目录项、选项或服务。

​		**变量集（Variable Sets）**：可在目录项之间共享的模块化变量单元。

​		**Order Guide：**将多个相关目录项（如 “新员工入职包”）组合成一个统一入口，用户一次提交即可完成多项申请。

​	关键价值：

​		**自助服务**：减少 IT 服务台的工单量，提升员工满意度。

​		**标准化**：确保所有服务申请都遵循统一的流程和定价。

​		**可扩展性**：通过多目录和视图功能，适应大型企业的复杂组织架构。



​	**Variable Types：**

​		Multiple Choice（多选 / 单选按钮）：创建单选按钮，供用户选择自定义问题选项。

​		Select Box（选择框）：创建下拉选择列表，供用户选择自定义问题选项。

​		Single Line Text（单行文本）：创建单行文本输入框。

​		Reference（引用字段）：指定另一张表中的一条记录，功能类似于引用字段。

​	请求项的进度阶段：

​		**等待审批（进行中）**（Waiting for approval (In Progress)）

​		**已批准**（Approved）

​		**待处理（尚未开始）**（Pending (has not started)）

​		**履行中（进行中）**（Fulfillment (In Progress)）

​		**已完成**（Completed）

#### 4.3 工作流（workflows）

​	工作流：工作流是对一系列活动的虚拟表示，这些活动由按顺序规划的连接步骤组成。

​	**何时使用工作流：**

​		用于自动化可重复的流程

​		当每条记录的插入、更新、删除和创建都需要标准响应时

​		当需要在平台中实现自动化时（不限于服务目录和 SLA）

​	**工作流启动方式：**

​		计划任务或 SLA 触发（Schedule or SLA launch）

​		表交互触发（满足条件时执行）

​		从脚本包含（Script Include）或业务规则（Business Rule）中调用

​		在工作流画布上点击 “Start”（仅管理员测试使用）

​		与服务目录（Service Catalog）中的目录项（Catalog Item）关联

​		从一个工作流中调用另一个工作流，该工作流被称为该工作流的**子工作流（subflows）**。

​	工作流状态：

​		已发布（Published）：可供所有用户和平台使用的工作流版本，满足工作流条件时可用于创建上下文（contexts）。

​		未发布（Unpublished）：不再用于新上下文的工作流版本，但对于已在运行的上下文可能仍需使用（Published = false）。**注意**：同一时间只能有一个工作流的已发布版本，过去的版本作为未发布版本保留，因此是历史记录。

​		已检出（Checked Out）：仅对检出该版本的用户可用的工作流。

​	工作流时间线显示进度（Workflow Timeline Shows Progress）和显示时间线（Show Timeline）提供了与工作流关联的活动的完整基于时间的显示。甘特图（Gantt Charts）用于显示此类时间线。使用时间线控件可以隔离特定的执行路径，或随时间比较多个执行路径。时间线会显示每个活动在彼此关系中的执行路径。你可以隔离执行路径，并跟踪子工作流与主流程之间的过渡线。处理时间提供了从工作流图中无法获取的工作流视图。工具提示（Tooltips）提供了每个活动的精确信息，例如持续时间。

​	一个工作流版本保持这些属性：

​		工作流表关联。

​		工作流的条件、发布状态和权限。任何时候只能有一个有效的版本。

​		工作流模型，即工作流的完整形式，作为工作流活动及其转换的集合。

​	为何需要工作流版本（Workflow Versions）：

​		为防止用户对会影响其他用户的工作流进行更改，工作流需要先被检出才能编辑。

​		当工作流被检出时，更改仅应用于检出该工作流的用户。其他用户可继续使用已发布的工作流。所有更改完成后，可发布工作流，使其对所有用户可用。

​	创建新工作流流程：

​		创建新工作流（Create New Workflow）

​		添加审批活动（Approval Activities）

​		添加目录任务（Catalog Tasks）

​		设置值（Set Values）

​		设置通知（Setup Notifications）

​		发布工作流（Publish Workflow）

#### 4.4 SLA

​	**服务级别协议（SLA）** 是 SLA（`contract_sla`）表中的一条记录，它定义了任务达到特定条件所需的时间量、要访问的表（s），以及要评估的 SLA 类型。

​	SLA有几个主要组件协同工作：

​		SLA 定义（SLA Definition）:定义触发 SLA 条件的记录。

​		任务 SLA（Task SLA）:与特定任务关联的 SLA 单个实例。

​		SLA 工作流（SLA Workflow）:基于 SLA 定义触发事件或操作的工作流。

​		SLA 条件与脚本包含（SLA Conditions and Script Include）:可用于自定义不同 SLA 状态之间过渡的脚本包含和引用记录。

​	SLA条件：启动，取消，暂停，回复，停止，重置条件。

​	SLA条件评估：

​		系统中的每个任务按一下属于内需进行评估:

​			处理新 SLA，以确定是否必须将新 SLA 记录附加到任务。

​			处理附加到任务的现有 SLA 记录。

​		**SLA条件**按以下顺序进行**评估**

​			**附加**：如果启动条件匹配，且停止和取消条件不匹配。

​			**完成**：如果停止条件匹配。

​			**暂停**：如果暂停条件匹配。

​			**恢复**：如果暂停条件不匹配，且恢复条件匹配。

​			**重新附加**：如果重置和启动条件都匹配。

​		**取消**：如果启动条件不匹配，且取消条件匹配

### 5.核心应用管理议程Core Application Administration Agenda

#### 5.1策略与交互（Policies and Interactions）

​	UI 策略（UI Policies）、数据策略（Data Policies）、UI 操作（UI Actions）、业务规则（Business Rules）和脚本（Scripts）都是用于定制 ServiceNow 实例和应用的强大工具。

​	ServiceNow 采用软件即服务（SaaS）模式：网页浏览器是客户端。客户端到服务器的往返会消耗时间，并让最终用户等待往返完成。这种等待是因为客户端脚本通常是同步执行的。

​	UI 策略与 UI 策略操作、数据策略与 UI 操作：

​		UI 策略（UI Policy）：

​			**运行位置**：客户端（Client Side）

​			**作用**：作为客户端脚本的替代方案，用于在表单上动态更改信息。

​			**示例**：使表单中的字段变为只读。

​			**特点**：一旦保存新的 UI 策略，相关列表中会显示 UI 策略操作。

​		UI 策略操作（UI Policy Action）

​			**运行位置**：客户端和服务器端（Client Side and Server Side）

​			**作用**：执行 UI 策略定义的行为，例如隐藏字段、设置字段为只读等。

​			**补充说明**：数据策略可以将字段设为必填或只读，而 UI 策略操作也可以隐藏字段。

​		数据策略（Data Policy）：

​			**运行位置**：客户端 / 服务器端（Client/Server Side）

​			**作用**：通过为字段设置必填和只读状态，强制数据一致性。

​			**示例**：在表单和导入时强制执行必填数据。

​			**特点**：与 UI 策略类似，但数据策略会应用于所有进入系统的数据，包括通过导入集或 Web 服务导入的数据，以及通过移动 UI 输入的数据；而 UI 策略仅应用于通过标准浏览器输入到表单的数据。

​		UI 操作（UI Action）：

​			**运行位置**：客户端 / 服务器端（Client/Server Side，取决于 “client” 复选框的选择）

​			**作用**：在表单和相关列表项上添加按钮、链接和上下文菜单项，创建交互式、可定制的特定用户活动。

​			**示例**：使 “保存” 按钮在表单上对相应用户可见。

​			**特点**：可以包含脚脚以定义自定义流程。

​	**Business Rule**：是一段 JavaScript 脚本，配置为在记录（Record）显示、插入、更新、删除时，或在查询表（Table）时运行

​	Business Rule可设置为在数据库操作发生之前或之后运行。

​	与 UI 策略不同，业务规则不监控表单上的字段（Field）。尽管在 ServiceNow 应用中有多种控制行为的方式，但大多数系统行为的自定义都是通过业务规则完成的。业务规则在用户与平台的每次交互开始时加载和初始化。全局业务规则可用于定义将被各种脚本（Script）重复使用的函数。

​	每个业务规则都包含要运行的表（Table）和时机（插入前 / 插入后等）、要评估的条件、基于评估结果要运行的脚本，以及是否支持客户端调用。

​	业务规则在服务器上运行，但可以支持客户端调用。如果勾选了 “客户端可调用（Client callable）” 设置，客户端可以使用 AJAX 调用该业务规则。

​	**Business Rule处理流程：**

​		用户或系统查询 → 查询规则（Query rules） → 数据库查询

​		表单提交 → 显示规则（Display Rules） → 前置规则（Before Rules） → 数据库更新 → 后置规则（After Rules）

​		无论通过表单、列表还是 Web Services 访问记录（Record），业务规则都会一致地应用。这是业务规则与客户端脚本（Client Script）的主要区别之一，后者仅在通过表单编辑时生效。

​	时机设置（When setting）允许你选择Business Rule应在何时执行：

​		记录（Record）保存到数据库之前

​		记录保存到数据库之后

​		异步（排队）；客户端与服务器独立工作，因此客户端无需等待服务器响应

​		记录显示之前（显示）

​	（可利用暂存区（scratchpad）访问服务器数据）

​	**全局业务规则（Global Business Rules）**

​		“全局（Global）” 是表（Table）字段中的一个设置，它**使业务规则（Business Rule）对所有表可用**

​		ServiceNow 强烈建议**不要**使用全局业务规则 —— 请改用脚本包含（Script Include）

​		标记为在 “全局（Global）” 表上运行的业务规则，会在用户与平台的每次交互开始时加载和初始化。

​		全局业务规则会**忽略 “条件（Condition）” 字段**，因此如果业务规则包含脚本，且脚本被封装在函数		中，那么它会在用户与平台的每次交互中运行。这是一种不良实践，会降低系统性能。

​		将脚本封装在函数中的全局业务规则，可以被其他任何地方运行的脚本调用。

​		业务规则（以及客户端脚本（Client Script））可以引用在全局业务规则中定义的任何函数。

​		全局业务规则可用于定义被多种脚本重复使用的函数。
